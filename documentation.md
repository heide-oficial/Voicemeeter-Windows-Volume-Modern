## Project Overview

Voicemeeter Windows Volume Modern keeps Windows output volume and mute changes synchronized with selected Voicemeeter strips and buses. It is a native Windows rewrite of the original Node.js tray app at `https://github.com/Frosthaven/voicemeeter-windows-volume`.

The current implementation uses C#/.NET, WinUI 3, Windows App SDK, Windows Core Audio callbacks, and direct Voicemeeter Remote calls. The legacy Node.js, PowerShell polling workers, `ffi-napi`, webpack, nexe, and NSIS runtime path have been removed from the app.

## Architecture

The solution is organized under `native/`:

- `VMWV.App` is the WinUI 3 desktop app. It owns navigation, pages, tray integration, title bar/icon handling, close-to-tray, and single-instance activation.
- `VMWV.Core` contains app settings, JSON persistence helpers, volume mapping rules, and service contracts.
- `VMWV.Infrastructure.Windows` contains Windows-specific implementations for Core Audio monitoring and Voicemeeter Remote integration.
- `VMWV.Core.Tests` is a lightweight console test runner for core logic and settings normalization.
- `VMWV.Portable` builds the portable EXE bootstrapper used for GitHub release assets.
- `VMWV.Installer` builds the native Windows Installer MSI used for GitHub release assets.

The app follows a simple service + view model split. `MainPageViewModel` coordinates audio service events, Voicemeeter state, settings persistence, diagnostic logs, and UI command state.

## Runtime Flow

On startup:

1. `App` registers a single app instance key and redirects later launches to the existing window.
2. `MainWindow` initializes the custom title bar, icon variant, tray hooks, window size, and root frame.
3. `MainPage` creates `MainPageViewModel`, then starts audio monitoring when loaded.
4. The Windows audio service reports the default endpoint snapshot and raises volume/mute/device callbacks.
5. The app attempts to connect to Voicemeeter automatically up to five times, ten seconds apart.
6. When connected, available Voicemeeter targets are loaded and the selected strip/bus bindings are used for synchronization.

Volume and mute events are coalesced before Voicemeeter updates. This avoids spawning one remote operation per rapid Windows audio event and keeps only the latest pending state.

## Settings and Persistence

Settings are stored as JSON at the user-local path defined by `AppSettingsPaths.DefaultSettingsPath`. The store uses source-generated JSON serialization and normalizes invalid or duplicate binding toggles when loading.

Most setting writes are debounced to reduce disk I/O. Settings that affect window lifecycle, such as close-to-tray, are saved immediately.

Important persisted options include:

- startup with Windows;
- close to tray;
- mute sync;
- remembered volume;
- volume mapping range and mode;
- Voicemeeter strip/bus binding toggles;
- recovery options;
- logo variant.

## UI Structure

The app uses a WinUI `NavigationView` with these pages:

- Dashboard - Voicemeeter status, Windows audio status, active strip/bus bindings, and recent logs.
- Bindings - selectable Voicemeeter strip and bus targets.
- Settings - control commands, appearance, startup/sync options, volume mapping, and advanced recovery options.
- About - modern project and original project attribution.

The UI uses native WinUI controls, theme resources, responsive grids, and compact Windows 11-style setting rows. Icon-only actions have automation names and tooltips.

## Release Packaging

Release artifacts are generated by `scripts/Create-ReleaseArtifacts.ps1`.

The release script publishes `VMWV.App` as an unpackaged, self-contained app folder, copies the project license into the payload, and builds two release artifacts:

- `VMWV.Portable`, which extracts the payload to a versioned temp directory and launches the app.
- `VMWV.Installer`, a WiX Toolset project that builds a per-user Windows Installer MSI with native directory and feature-selection UI. The optional installer features are Start Menu shortcut and desktop shortcut.

```powershell
rtk powershell -NoProfile -ExecutionPolicy Bypass -File scripts\Create-ReleaseArtifacts.ps1 -Platform x64
```

Output files:

- `artifacts\release\VoicemeeterWindowsVolumeModern-Portable-x64.exe`
- `artifacts\release\VoicemeeterWindowsVolumeModern-Setup-x64.msi`

Generated artifacts are ignored by Git and should be attached to GitHub Releases instead of committed.

## Development Workflow

Build:

```powershell
rtk dotnet build VMWV.slnx -c Debug
rtk dotnet build VMWV.slnx -c Release
```

Run core tests:

```powershell
rtk dotnet run --project native\VMWV.Core.Tests\VMWV.Core.Tests.csproj
```

Packaged WinUI development runs still use WinApp CLI. Release artifacts use the self-contained publish path.

## Validation

Before publishing a release:

1. Build Debug and Release.
2. Run `VMWV.Core.Tests`.
3. Generate release artifacts.
4. Launch the portable EXE directly.
5. Install using the MSI setup package.
6. Confirm single-instance behavior by launching the app twice.
7. Confirm Voicemeeter connection, strip/bus bindings, tray behavior, settings persistence, and uninstall.
8. Confirm `git status --short` contains only intentional source/documentation changes and no generated artifacts.

## Operational Notes

- Voicemeeter must be installed and running for synchronization.
- The app detects common Voicemeeter process names and broader Voicemeeter executable-name variants.
- Diagnostics retain a bounded set of recent entries and collapse repeated volume-change events by updating the latest log row.
- Close-to-tray keeps the app running in the background; exiting from the tray menu closes it.
